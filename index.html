ABSH

Motive

To build and understand the BASH scripting language's workings and become comfortable programming in multiprocess environments. ABSH is a play on BASH, it is based on my initials followed by SHell.

Developement

The entire project was built in C. It contains a REPL which reads user input, parses it and checks for the commands in the builtin commands located in the user's bin folder. If these commands are not present, execv is called within a child process. I also had to manage the user's jobs, which was done using a Linked List.

Features

The shell can be run on any terminal and contains extensive functionality, such as job control (users can track processes and move them between the foreground and background at will).

The shell also allows for file redirection (for input and output).

Database

Motive

To create a program that can interact with a database efficiently and securely through the use of threads.

Developement

The database runs on a server which can be accessed by any number of users at a time who can peform actions simultaneously through the use of threads. It is implemented as a Binary Search Tree. Each node has a key value pair, making the database a dictionary.

The server has a listener thread that is the only thread to respond to signals. It also initiates all the connections with potential clients and interprets/executes all the client commands. It is also responsible for managing the number of clients and locks and unlocks mutexes accordingly.

There is another file responsible for the execution of the manipulation and querying of the database. Here, read and write locks are used as to provide for greater efficiency when traversing the database for independent purposes.

Features

The database is made thread safe by the use of mutexes. Rather than locking the entire database when a user requests to manipulate data, the database employs fine grained locking, where only the elements that are crucial do the manipulation are locked.

The client programs can add (a), delete (d) and query (q) the database. The database specific functions all rely on a search function that is also completely thread safe.

The server can print (p <optional output file>) the database, inhibit clients from interacting with the database (s), release clients to interact with it once again (g) and finally if an interrupt signal is sent to the server, all the connections between the current clients and the server is terminated and new clients can connect.

Search

Motive

Explore writing programs that work with large datasets without losing efficiency or functionality.

Background & Development

Up until Google, most search engines gave the user results based only on the relevance to the query. Today, most search engines have a ranking system which combines the relevance of the page to the query with its relative popularity to the other online pages. This search engine has both implementations.

The entire project was built in Scala. The main challenge of this project was to parse through thousands of pages without wasting the users time waiting 5 minutes for a search result to be displayed. This was done by preparsing every page and creating a dictionary in which we store the relevance of the page to the user's query.

Features

The browser can load some simple online pages but it is primed for pages we created, which are complete with forms and dynamic elements (that respond to user actions and input).

Memphis Taxis AB Testing & Eyetracking

Motive

To prototype two different websites and use methods of AB Testing and eyetacking software to create a hypothesis about which version is more favorable in terms of usability principles.

Developement

Two websites were built using HTML and CSS. We then used Heroku to track how users used our website. We were able to determine when the user clicked on links, left our website and returned.

From just this data, we were able to calculate Click Through Rate, Time to Click, Dwell Time and Return Rate. I then used a combination of T-Test and Chi-Square Test to determine if one of the websites was statistically significantly "better" than the other.

Results

After performing (lots of) calculations, we found that users took statistically significant less time to click in version A compared with Version B. We equated the Time to Click with ease of navigation and finding information.

After analyzing the eyetracking results (which we made into a heatmap and into a real time visual drawing of the gaze of the user using Javascript), we found that there was less wandering in Version B compared to Version A. A further discussion can be found in the complete report, which can be found here.

Dropleaf Iterative Redesign

Motive

To prototype an entire web application for a startup to get experience with designing as a team.

Background

Dropleaf is a startup that aims to create a space for gamers to play indie video games on a monthly subscription basis, similar to services such as Netflix and Hulu. You can find their websie here.

Developement

My team of four started off creating individual sketches and discussing trade offs between our design choices. We then moved on to creating high fidelity prototypes of what we considered important pages for the Dropleaf website (which included the home page, the browse page, settings and more).

Results

We reached out to Dropleaf and they said they were very impressed and satisfied with our mockup. A complete report on the design process can be found here.

Design Choices

The design choices we made were based primarily on two things. The first was we wanted to emulate as much as possible Netflix's interface since Dropleaf frequently refers to themselves as Netflix for indie Video Games. We assumed that Dropleaf users are most likely also Netflix users, so we wanted to make the interfaces navigation very similar to make the adaptability to Dropleaf smooth.

The rest of our design choices relied heavily on Heuristic Evaluation to detemrine what was user centered design or not. Thus, we created a color palette prior to the building of the interface, we went through some serial wireframing to arrive at a consensus, we used a same 12 column grid for every page and used drop shadow, rounded edges and other material design elements to reduce the cognitive load on the user.

Seabrooktx.gov Responsive Redesign

Motive

Creating a responsive redesign of a current website lacking in user centered design.

Developement

I selected 5 pages of the official website for Seabrook, Texas and redesigned them using Jacob Nielsen's 10 Heuristic Usability Principles. I then moved on to creating a high fidelity mockup of one of them using Adobe Photoshop and finally created a responsive HTML and CSS website of the government page.

Details

The main tactic to make the website responsive was using @media tags to create styling (CSS) changes to the website without having to create a new version of it. These changes include less images and a cleaner, more simple layout since users who access the site on a mobile device are likely to want to spend less time navigating than a user on a desktop. The final and complete report can be read here.

GUIzilla

Motive

To understand the principles of network programming and Graphical User Interfaces

Developement

The backend of the project includes a Scala built Client and Server which communicate with each other based on the standard protocol requests and responses. The front end is built on JavaFX with a controller that is responsible for loading new pages with the help of an FXML parser also built in Scala.

Features

The browser can load some online pages but it is primed for pages we created as well, which are complete with forms and dynamic pages (that respond to user actions and input)

Another feature we added to our Browser is the presence of a back button, created using a queue that stores the elements of pages after users have entered their information. This is a feature that is actually not deployed by many browsers. On GUIzilla, users are able to see the web page exactly as they left it, with their inputs and changes present.
